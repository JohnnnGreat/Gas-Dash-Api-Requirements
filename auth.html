Gas Delivery Application - Backend Requirements

Date: June 15, 2023

Version: 1.0

This document outlines the backend requirements for the gas delivery mobile application's authentication system, including database schema, API endpoints, and technical specifications.

1. Database Schema

1.1 Users Table

CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    google_id VARCHAR(255) UNIQUE,
    full_name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    phone_number VARCHAR(20),
    password_hash VARCHAR(255),
    email_verified BOOLEAN DEFAULT FALSE,
    phone_verified BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    last_login TIMESTAMP WITH TIME ZONE,
    status VARCHAR(20) DEFAULT 'active' CHECK (status IN ('active', 'suspended', 'deleted'))
    user_type VARCHAR(20) DEFAULT 'customer' CHECK (status IN ('wholesaler', 'driver', 'filling_station'))
    profile_picture_url VARCHAR(255),
    
);

CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_phone ON users(phone_number);

1.2 Password Reset Tokens Table

CREATE TABLE password_reset_tokens (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id) ON DELETE CASCADE,
    token VARCHAR(255) NOT NULL,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    used BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_password_reset_tokens_token ON password_reset_tokens(token);

1.3 Login Sessions Table

CREATE TABLE login_sessions (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id) ON DELETE CASCADE,
    session_token VARCHAR(255) NOT NULL,
    device_info TEXT,
    ip_address VARCHAR(45),
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_login_sessions_token ON login_sessions(session_token);
CREATE INDEX idx_login_sessions_user ON login_sessions(user_id);

2. API Endpoints

2.1 Authentication Endpoints

POST /api/auth/register

Description: Register a new user account

Request:

{
  "full_name": "John Doe",
  "email": "john@example.com",
  "phone_number": "0912441283",
  "password": "securePassword123!",
  "confirm_password": "securePassword123!"
}

Response (Success - 201 Created):

{
  "success": true,
  "data": {
    "id": 123,
    "email": "john@example.com",
    "full_name": "John Doe",
    "phone_number": "0912441283",
    "email_verified": false
  },
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}

Response (Error - 400 Bad Request):

{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Password must be at least 8 characters long",
    "details": {
      "password": "Password must be at least 8 characters long"
    }
  }
}
POST /api/auth/login

Description: Authenticate an existing user

Request:

{
  "email": "john@example.com",
  "password": "securePassword123!"
}

Response (Success - 200 OK):

{
  "success": true,
  "data": {
    "id": 123,
    "email": "john@example.com",
    "full_name": "John Doe",
    "phone_number": "0912441283",
    "email_verified": true
  },
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}

Response (Error - 401 Unauthorized):

{
  "success": false,
  "error": {
    "code": "INVALID_CREDENTIALS",
    "message": "Invalid email or password"
  }
}

3. Technical Specifications

3.1 Security Requirements

  • Password Hashing: Use bcrypt with cost factor 12
  • JWT Tokens:
    • Algorithm: HS256
    • Access Token Expiry: 1 hour
    • Refresh Token Expiry: 7 days
  • Rate Limiting:
    • 5 requests per minute for /auth endpoints
    • 100 requests per minute for authenticated endpoints
  • CORS: Restrict to mobile app domains only
  • Input Validation: Validate all inputs on server side

3.2 Password Requirements

  • Minimum 8 characters
  • At least one uppercase letter
  • At least one lowercase letter
  • At least one number
  • At least one special character

5. Error Handling

Standard error response format:

{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message",
    "details": {
      "field1": "Error detail for field1",
      "field2": "Error detail for field2"
    }
  }
}

Common Error Codes:

  • VALIDATION_ERROR
  • INVALID_CREDENTIALS
  • EMAIL_ALREADY_EXISTS
  • UNAUTHORIZED
  • TOKEN_EXPIRED
  • USER_NOT_FOUND

Note: The backend should provide OpenAPI/Swagger documentation, Postman collection, and example requests/responses for all endpoints.
